# A z-index child inside a fixed, stacking-context ancestor can never paint above a sibling ancestor; interactive elements meant to sit "in the sky" need their own top-level layer.

Confirmed while building the drag-the-sun gimmick (2026-07-02): `.sky-container` is `position: fixed; z-index: 0`, which creates a stacking context, and `main` is `position: relative; z-index: 1`. The sun/moon (`.celestial__body`) live inside `.sky-container`. Giving them `pointer-events: auto` and a `cursor: grab` did nothing: `document.elementFromPoint()` at the sun's exact screen coordinates resolved to `.threshold` (an empty, non-interactive spacer inside `main`), because no z-index value on a descendant can escape its ancestor's stacking context. `main`'s content, even invisible spacer sections, always intercepted the pointer first.

Fix: added a separate invisible layer (`CelestialGrab.astro`), `position: fixed; z-index: 2` (above `main`), rendered as a sibling after `<main>` in `BaseLayout.astro`, tracking the same `--celestial-x/y` CSS vars so it sits exactly on top of the visible sun/moon without changing any pixel of the visual design. The visible sun/moon in `Sky.astro` stayed pointer-events: none and purely decorative; all drag logic (pointerdown/move/up, grab-target gating by `--sun-opacity`, double-tap reset) moved to the new layer.

General rule: never assume `pointer-events: auto` + a script listener is enough to make something in a `position: fixed` background layer clickable if the page has separately-stacked foreground content on top. Check with `elementFromPoint` at the actual screen coordinates before assuming a click handler will fire.
